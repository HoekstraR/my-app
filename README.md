## Security defaults

These are baked into the project so you don't need to reinvent them:

- **`react/no-danger` lint rule** — `dangerouslySetInnerHTML` is forbidden by default. AI-generated apps usually render external data; raw HTML injection is almost never intentional. If you have a genuine need, opt out per-file in `eslint.config.mjs`.
- **HttpOnly cookies for auth tokens** — any OAuth token Walden wires up (Supabase, GitHub, etc.) is stored as `HttpOnly`, `SameSite=Lax`, `Secure` in production. JavaScript on the page cannot read it, so an XSS bug cannot exfiltrate the token.
- **Supabase RLS-first patterns** — when Dana (Walden's data engineer) creates user-facing tables, she enables Row-Level Security and writes policies scoped to `auth.uid()` by default. Public-anon access is opt-in, not the default.
- **Migrations are append-only** — every schema change Dana applies lives in `database/migrations/<timestamp>_<name>.sql`. The Supabase project is the live state; the migration files are your source-controlled history.

## How Walden built this

Walden is a multi-agent system. An orchestrator routes your requests to specialists:

- **Dex** writes application code (components, routes, logic).
- **Dana** owns the database (Supabase migrations, RLS, types).
- **Derik** debugs failures.
- **Didi** handles design and styling.

Each specialist has a constrained surface — Dana can only write to `database/migrations/`, for example. This is why the project has a clean separation between code and schema, and why migrations are never inlined into application files.

## File conventions

- **`database/migrations/*.sql`** — schema history. Never edit applied migrations; add a new one instead.
- **`types/database.ts`** — TypeScript types generated from your live Supabase schema. Regenerate after every migration via Supabase's `generate_typescript_types`.
- **`.env.local`** — secrets and Supabase URL. Not committed.
